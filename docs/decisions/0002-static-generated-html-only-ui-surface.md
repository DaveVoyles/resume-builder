# Static generated HTML remains the only UI surface

- Status: accepted
- Date: 2026-07-20
- Deciders: Dave Voyles

## Context and Problem Statement

Design plan 0005 adopts several concepts from scottgal/lucidRESUME, a
C#/Avalonia *desktop* application whose UI (nine pages: pipeline funnel,
skills dashboard, DOCX preview, job boards) is a large part of its appeal.
Adopting its UI ideas raised the question: should this project grow a real
graphical interface — a served local web dashboard, an SPA, or a desktop
app — or keep its current shape, where the only visual surface is a static
HTML file the CLI regenerates?

## Decision Drivers

- ADR 0001 commits the CLI to being deterministic, LLM-free, and
  API-key-free; a served UI invites state, processes, and complexity that
  erode that stance.
- The tool is agent-operated: the terminal agent, not a GUI, is the
  primary interface. A second interactive surface would compete with it.
- A generated file needs no server lifecycle, no ports, no security
  surface, and works identically on every machine the workspace syncs to.
- The existing HTML tracker already proves the generated-file pattern can
  carry dashboards (stat cards, search, filters).

## Considered Options

1. Static generated HTML only (richer dashboard within the existing
   build-tracker output)
2. Local served web dashboard (small HTTP server + live views)
3. Desktop app (port or imitate lucidRESUME's Avalonia approach)

## Decision Outcome

Chosen option: **static generated HTML only**. UI investment goes into the
generated tracker file — funnel view, stale badges, card-based visual
refresh (plan 0005 D3/D4) — which stays a single self-contained artifact
regenerated by `build-tracker`, openable from disk with no server.

### Consequences

- Good: zero new runtime surface — no server, ports, sessions, or
  security considerations; the ADR 0001 architecture is untouched.
- Good: the dashboard is versionable, diffable, portable output — same
  properties as every other rendered artifact (DOCX, markdown).
- Bad: no live interactivity beyond what client-side JS in a static file
  can do (no push updates; the file must be regenerated to refresh).
- Bad: some lucidRESUME UI ideas (live DOCX preview, in-app editing) are
  permanently out of reach in this shape; revisiting them requires
  superseding this ADR.
